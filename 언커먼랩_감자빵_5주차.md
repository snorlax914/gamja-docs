## 1. 통합 MVP 서비스 구현

### 1.1 처리 흐름 개요

```
[파일 업로드]
      │
      ▼
[FastAPI: 파일 저장 + 백그라운드 태스크 시작]
      │
      ▼
[① OCR]  PaddleOCR-VL (vLLM 서버, :8118)
      │  → PDF는 pdf2image로 페이지 분할
      │
      ▼
[② 분류]  Ollama / qwen3:4b
      │  → 10개 카테고리 중 하나로 JSON 강제 출력
      │
      ▼
[③ 청킹]  마크다운 인지 청커 (5주차에 새로 도입)
      │  → 헤더 경로 보존 + 표/코드 보호
      │
      ▼
[④ 임베딩]  BGE-M3 (CPU)
      │
      ▼
[⑤ Qdrant 색인]  doc_id + 카테고리 메타데이터
      │
      ▼
[준비 완료]  대시보드에 전체 마크다운/청크 표시 + RAG 챗 활성화
```

각 단계 진입 시점에 `doc_store` 의 `status` 필드가 `processing → classifying → indexing → ready` 로 갱신되고, 프론트엔드 폴링(1.5초 간격)이 이를 감지해 진행 인디케이터를 업데이트한다.

### 1.2 백엔드 (FastAPI)

#### 1.2.1 신규 엔드포인트

5주차 요구사항(“결과 대시보드에서 OCR로 추출된 전체 마크다운과 쪼개진 Chunk를 사용자가 직접 확인할 수 있어야 한다”)을 만족하기 위해 두 개의 읽기 전용 엔드포인트를 추가했다.

| 메서드 | 경로 | 응답 모델 | 설명 |
|---|---|---|---|
| GET | `/documents/{doc_id}/markdown` | `MarkdownResponse` | OCR로 생성된 전체 마크다운 원문 |
| GET | `/documents/{doc_id}/chunks` | `ChunksResponse` | 동일한 청커로 재산출한 청크 목록 + `chunk_size`/`chunk_overlap` 메타 |

전체 마크다운은 업로드 시 함께 저장해 둔 `uploads/{doc_id}.txt` 에서 읽고, 청크는 해당 원문을 같은 청커로 다시 잘라 반환한다. Qdrant에 이미 청크가 들어가 있지만 결정론적인 재계산이 더 단순하고, 청커 튜닝 시 화면 결과가 바로 반영된다는 장점이 있다.

기존 API(`POST /documents/upload`, `GET /documents`, `DELETE /documents/{id}`, `POST /chat` (SSE))는 그대로 유지하여 호환성을 보장했다.

#### 1.2.2 응답 모델 (`backend/app/schemas.py`)

```python
class MarkdownResponse(BaseModel):
    doc_id: str
    markdown: str
    length: int

class ChunkInfo(BaseModel):
    idx: int
    text: str
    length: int

class ChunksResponse(BaseModel):
    doc_id: str
    count: int
    chunk_size: int
    chunk_overlap: int
    chunks: list[ChunkInfo]
```

`DocumentInfo` 에는 진행 인디케이터 표시용으로 `full_text_length` 필드를 추가했다.

#### 1.2.3 단계별 소요 시간 로깅

`_process_document` 백그라운드 태스크는 OCR / 분류 / 청킹 / 임베딩 / 색인 각 단계의 소요 시간을 `timings` 사전에 모으고 완료 직후 한 줄 요약 로그로 출력한다. 추후 병목 분석(예: PDF 다중 페이지 OCR이 전체의 몇 %를 차지하는가)에 그대로 활용할 수 있다.

### 1.3 프론트엔드 (Next.js 15)

#### 1.3.1 화면 구성

```
┌────────────────────────────────────────────────────────────────┐
│  doc·rag                                                       │
├────────────────────┬───────────────────────────────────────────┤
│  UploadZone        │  DocDetail (분류 결과 · 상태 뱃지)         │
│  ──────────────    │  ResultDashboard (5주차에 신규 추가)       │
│  DocList           │    · pipeline 진행 인디케이터              │
│   · 상태 점멸 인디케이터 │    · 탭 1: full markdown (원문/raw)     │
│   · 분류 카테고리   │    · 탭 2: chunks (접고 펴기 가능)         │
│                    │  ChatPanel (SSE 스트리밍 RAG)              │
└────────────────────┴───────────────────────────────────────────┘
```

#### 1.3.2 ResultDashboard 컴포넌트

신규 컴포넌트 `frontend/components/ResultDashboard.tsx` 가 다음을 담당한다.

1. **파이프라인 인디케이터** — OCR → 분류 → 청킹·임베딩·색인 → 완료의 4단계를 좌측부터 채워나가는 가로 스텝퍼. 현재 단계는 호박색 펄스, 완료 단계는 에메랄드 ✓, 오류는 적색.
2. **전체 마크다운 뷰** — 두 가지 모드를 토글로 전환.
   - `rendered`: 헤더/표/코드 블록/리스트/수평선을 직접 렌더링한 경량 마크다운 뷰. 외부 의존성 없이 직접 구현해 번들 크기를 늘리지 않았다.
   - `raw`: `<pre>` 로 원문 그대로 표시.
3. **청크 뷰** — 청크별 카드(번호·길이·미리보기 한 줄)를 보여주고, 클릭하면 전체 텍스트로 펼쳐진다. 헤더에 청크 수 / 평균 길이 / 총 글자 수 / `chunk_size` / `overlap` 메타를 함께 표시.

처리 상태가 `ready` 가 되었을 때만 `/documents/{id}/markdown` 와 `/documents/{id}/chunks` 를 `Promise.all` 로 병렬 조회해 한 번에 그린다.

#### 1.3.3 처리 상태 인디케이터

3 레이어로 사용자에게 진행 상황을 보여준다.

| 위치 | 표현 |
|---|---|
| 좌측 문서 리스트 | 작은 점 인디케이터 + “처리 중 · OCR/분류/색인” 텍스트 (펄스 애니메이션) |
| 상단 DocDetail | 상태 뱃지 (호박색 펄스 · 에메랄드 · 적색) |
| 대시보드 상단 | 4단계 가로 스텝퍼 |

폴링은 `DocList` 에서 “진행 중 문서가 하나라도 존재”할 때만 1.5초 간격으로 동작하고, 모두 `ready`/`error` 가 되면 자동 중단된다.

### 1.4 통신 인터페이스

CORS 회피를 위해 모든 호출은 Next.js의 `rewrites` 프록시(`/api/* → http://localhost:8000`)를 통해 백엔드로 전달된다. RAG 답변은 SSE(Server-Sent Events)로 스트리밍하며, `EventSource` 가 POST 를 지원하지 않으므로 `fetch + ReadableStream` 으로 직접 파싱한다 (`frontend/lib/api.ts: chatStream`).

## 2. 핵심 기술 고도화 — 마크다운 인지 청킹

### 2.1 4주차 회고: 단순 문단 청킹의 한계

4주차의 청커(`chunk_text` 구버전)는 `\n\n` 단위로 자른 뒤 길이 기준 합치는 방식이었다. PaddleOCR-VL 출력에는 다음과 같은 구조가 흔히 등장하는데, 단순 문단 청킹은 이들을 검색 품질을 떨어뜨리는 방식으로 잘라버렸다.

1. **표(`|...|`)** 가 행 단위로 분리되어 헤더 행과 데이터 행이 다른 청크에 들어감.
2. **코드 펜스(```\`\`\`...\`\`\````)** 가 도중에 잘려 LLM이 컨텍스트로 받았을 때 코드라는 신호를 잃음.
3. **헤더(`# / ## / ###`)** 가 본문과 분리되어 “이 청크가 어느 절의 내용인지”를 LLM이 추론할 수 없음.

### 2.2 신규 청커 설계

`backend/app/services/chunker.py` 를 마크다운 인지 방식으로 전면 교체했다.

**1단계: 토큰화 (`_split_into_blocks`)**
원문을 라인 단위로 훑어 `header`, `code`, `table`, `para` 4종 블록으로 분리한다. `code`/`table` 은 `atomic=True` 로 표시해 절대 가운데서 자르지 않는다.

**2단계: 헤더 경로 그룹핑 (`_group_by_heading`)**
헤더 스택을 유지하면서 본문 블록을 누적한다. 새 헤더를 만나면 그 시점까지의 누적을 한 절(section)로 확정한다. 각 절은 `(헤더_경로, 본문)` 튜플로 표현되며, 헤더 경로는 청크 앞에 그대로 붙여 “이 청크가 ‘# 계약서 / ## 제 2 조 (목적)’ 안의 내용임”을 LLM이 즉시 알 수 있게 한다.

**3단계: 절 내부 재분할 (`_split_section`)**
절이 `chunk_size` 보다 크면 빈 줄 단위로 자르고, 그래도 큰 문단은 한/영 문장 분리로 폴백한다. 인접 청크 간에는 `chunk_overlap` 만큼 꼬리를 공유한다.

### 2.3 동작 예시

다음과 같은 마크다운을 `chunk_size=200, overlap=30` 으로 청킹하면:

```markdown
# Page 1

## 제 1 조 (목적)

이 계약은 ... 목적으로 한다.

## 제 2 조 (정의)

| 항목 | 의미 |
|---|---|
| 갑 | 회사 |
| 을 | 직원 |
```

`chunk 0` 은 `# Page 1 / ## 제 1 조 (목적) + 본문`, `chunk 1` 은 `# Page 1 / ## 제 2 조 (정의) + 본문 + 표 전체` 로 분리되어, **표가 한 청크 안에서 헤더 경로와 함께 보존**된다. 검증 스크립트로 확인한 결과 표가 행 단위로 쪼개지는 4주차의 문제는 재현되지 않는다.

### 2.4 영향과 후속 과제

| 항목 | 4주차 | 5주차 (현재) |
|---|---|---|
| 표가 청크 가운데에서 잘리는 빈도 | 빈번 | 0 (구조상 불가) |
| 청크에 절 컨텍스트 포함 여부 | 없음 | 헤더 경로가 prefix로 항상 포함 |
| LLM의 “문서 어느 부분인지” 인용 정확도 | 낮음 | 헤더 경로 덕에 향상 |
| 청크 길이 표준편차 | 큼 | 절 경계로 묶여 다소 감소 |

추후 과제(6주차 이후 후보):
- **하이브리드 검색**: Qdrant sparse + dense 결합으로 키워드/숫자 일치 강화.
- **재순위(rerank)**: BGE-reranker로 top-k 재정렬.
- **카테고리 필터**: 분류 결과(`category_code`)를 검색 단계에 필터로 주입해 잘못된 문서로의 유출 방지.
- **doc_store DB 화**: 현재 JSON 파일을 PostgreSQL 로 교체.

## 3. 시연 시나리오 (확인용)

1. `docker compose up -d` 로 Qdrant 기동, `ollama serve` 및 PaddleOCR-VL vLLM 컨테이너 기동, `uvicorn app.main:app --port 8000` 으로 백엔드 기동, `npm run dev` 로 프론트엔드 기동.
2. 브라우저에서 `http://localhost:3000` 접속.
3. 좌측 업로드 영역에 PDF/이미지 드롭 → 좌측 리스트에 펄스 인디케이터로 처리 단계 표시.
4. 우측 상단 DocDetail 의 상태 뱃지가 `OCR 진행 중 → 분류 진행 중 → 청킹·임베딩·색인 진행 중 → 준비 완료` 순으로 전환되는지 확인.
5. ResultDashboard 의 파이프라인 스텝퍼가 단계별로 ✓ 로 채워지는지 확인.
6. `full markdown` 탭에서 헤더/표가 살아 있는지 (rendered/raw 토글) 확인.
7. `chunks` 탭에서 각 청크 앞에 헤더 경로가 prefix 로 붙어 있고 표가 가운데서 잘리지 않았는지 확인.
8. 하단 ChatPanel 에서 질문 → SSE 스트리밍으로 답변과 `sources` 출처 청크가 함께 표시되는지 확인.
