## 1. 버그 수정 — LLM 분류 필드 누락 대응

### 1.1 발생 현상

`GET /documents` 호출 시 500 에러가 발생했다.

```
pydantic_core._pydantic_core.ValidationError: 1 validation error for DocumentInfo
classification.reason
  Field required [type=missing, ...]
```

### 1.2 원인 분석

`Classification` 스키마(`backend/app/schemas.py`)는 `reason: str`을 필수로 요구하지만, qwen3:4b 모델이 분류 JSON을 생성할 때 `reason` 키를 누락하는 경우가 발생했다.

- **qwen3:4b 모델 한계**: 소형 모델은 JSON 스키마 키를 간헐적으로 누락. thinking 단계에서 "분류 완료"로 판단하면 부가 필드를 최종 출력에서 빼먹는 경향이 있음.
- **프롬프트 모호**: `CLASSIFY_PROMPT`에 `reason` 필드가 "필수"라는 명시가 없었음.
- `OllamaService.classify()`는 JSON 파싱 성공 시 dict를 그대로 반환 → 누락된 채 `_docs.json`에 저장됨 → 이후 `DocumentInfo(**d)` 변환 시 검증 실패.

### 1.3 적용한 수정

`backend/app/schemas.py` — `reason`을 `Optional`로 완화.

```python
class Classification(BaseModel):
    category: str
    category_code: str
    confidence: float
    reason: Optional[str] = ""   # ← 기존: reason: str
```

기존 저장된 문서도 정상 동작하며, 향후 LLM이 동일 패턴으로 누락해도 500이 아닌 정상 응답이 가능하다.

### 1.4 식별된 유사 위험 (후속 작업 권장)

| 위험도 | 항목 | 내용 | 권장 수정 |
|---|---|---|---|
| High | `filename=None` 저장 가능 | `UploadFile.filename`이 `Optional[str]` → `None`이면 list 호출 시 500 | `filename=file.filename or "unnamed"` |
| High | `classify()` 다른 키 누락 가능 | `category`/`category_code`/`confidence`도 빠질 수 있음 | `.get()`으로 모든 필수 키에 기본값 보강 |
| Mid | `vector_store.py` KeyError | `r.payload["text"]`가 `[]` 접근 → 외부 손상 시 500 | `.get("text", "")`로 통일 |
| Mid | `_docs.json` 스키마 진화에 취약 | 새 필드 추가 시 기존 문서 전부 깨짐 | 새 필드는 `Optional`로 시작 또는 검증 실패 문서 스킵 |

> 자세한 분석은 `docs/week6/bug-audit-2026-05-16.md` 참조.

## 2. HTML 표 → 마크다운 변환

5주차 멘토 미팅의 결정 사항(“OCR 모델 옵션이 없으면 HTML → Markdown 변환 방식으로 신규 설계”)을 반영했다.

### 2.1 PaddleOCR-VL 옵션 확인

`paddleocr 3.5.0` / `paddlex 3.5.2` 의 내부 코드를 직접 추적한 결과, 표 인식 결과는 항상 HTML `<table>` 단편으로 생성됨을 확인했다.

- `paddlex/inference/pipelines/layout_parsing/result_v2.py:211-219` 의 `format_table_func` 는 `pretty=True`/`False` 모두에서 `<table>` 그대로 두고 외피 태그만 다듬는다.
- `_to_markdown(pretty=False)` 가 호출하는 `simplify_table()`(`markdown_format_funcs.py:208-226`)도 `<html>/<body>` 래퍼만 제거할 뿐 마크다운 변환은 수행하지 않는다.
- API 시그니처에 노출된 `format_block_content` 파라미터는 paddlex 3.5.2 내부에서 실제 사용되는 경로가 없다(grep 결과 0건).

→ 라이브러리 옵션으로는 마크다운 표를 얻을 수 없으므로, 자체 후처리 레이어를 추가하는 방향으로 결정했다.

### 2.2 변환 모듈 추가 (`backend/app/services/table_converter.py`)

`html_tables_to_markdown(text: str) -> str` 함수가 OCR 출력 안에 섞여 들어온 `<table>...</table>` 단편을 모두 GitHub-Flavored Markdown 표(`|...|`)로 치환한다.

| 처리 항목 | 규칙 |
|---|---|
| 헤더 행 판정 | `<thead>` 존재 시 그 행, 없으면 `<th>` 만 있는 행, 둘 다 없으면 첫 행을 헤더로 승격(GFM 표는 헤더 필수) |
| `rowspan` / `colspan` | 같은 셀 값을 펼쳐 직사각형 격자로 평탄화 |
| 셀 안 `<br>` | 공백으로 치환 |
| 셀 안 `\|` 문자 | `\\\|` 로 이스케이프 (표 구조 보호) |
| 연속 공백/줄바꿈 | 한 칸으로 정규화 |
| 변환 실패 | 원문 유지 (정보 손실 방지) |

`BeautifulSoup` 으로 파싱하되, 외부 의존성 추가를 명시하기 위해 `requirements.txt` 에 `beautifulsoup4==4.14.3` 을 직접 추가했다(기존엔 `paddleocr` 전이 의존성으로만 들어와 있어 깨질 위험이 있었음).

### 2.3 OCR 파이프라인 통합

`backend/app/services/ocr.py:_predict_image()` 끝에서 PaddleOCR-VL 의 `markdown_texts` 출력을 `html_tables_to_markdown()` 으로 한 번 더 거치도록 한 줄을 추가했다. PDF 의 모든 페이지가 동일 경로를 거치므로 전체 OCR 결과에서 표가 일관되게 마크다운으로 정규화된다.

### 2.4 효과

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 표 출력 형식 | `<table><tr><td>...` HTML | `\| 헤더1 \| 헤더2 \|\n\|---\|---\|\n\| ... \|` |
| 청커 인식 | `TABLE_ROW_RE` 미매칭 → 문단으로 잘림 | 매칭됨 → `atomic` 표 블록으로 한 청크에 보존 |
| 결과 대시보드의 `rendered` 모드 | 표가 HTML 로 표시 | 표가 마크다운 표로 깔끔하게 렌더링 |

5가지 입력 패턴(헤더 없는 단순 표 / `thead`+`tbody`+`th` / `rowspan`+`colspan` / `<br>`·파이프 포함 / 표 없는 입력 = no-op)에 대해 동작을 확인했다.

## 3. 다중 문서 질의 기능

5주차 미팅에서 식별된 “채팅이 전체 문서를 대상으로 가져오도록 되어 있음” 이슈와 6주차 과제 항목(“전체 문서를 대상으로 하는 채팅 서비스 구현”)을 함께 해결하기 위해, **전체/단일 이원 토글이 아니라 사용자가 임의의 문서 부분집합을 골라 질의할 수 있는 다중 선택 방식**으로 설계했다. 0개 선택은 자동으로 전체 모드가 된다.

### 3.1 백엔드

**`backend/app/schemas.py`** — `ChatRequest.doc_id: str` 을 `doc_ids: Optional[list[str]] = None` 으로 교체. `SearchHit` 에 `doc_id`, `filename` 추가.

**`backend/app/services/vector_store.py`** — `search()` 의 필터 분기.

```python
if doc_ids:
    if len(doc_ids) == 1:
        # 단일 — MatchValue
        query_filter = Filter(must=[FieldCondition(key="doc_id", match=MatchValue(value=doc_ids[0]))])
    else:
        # 다중 — MatchAny (합집합)
        query_filter = Filter(must=[FieldCondition(key="doc_id", match=MatchAny(any=list(doc_ids)))])
# 비어 있으면 필터 없음 → 전체 컬렉션 대상
```

**`backend/app/routers/chat.py`** — 다음을 추가했다.

1. `doc_ids` 정규화: 중복 제거, 입력 순서 보존.
2. 선택된 문서들의 존재/ready 상태 사전 검증 (단일/다중 공통).
3. 다중 또는 전체 모드에서는 LLM 컨텍스트 각 발췌 앞에 `[문서: <파일명>]` prefix 부착 → LLM 이 답변에서 어느 문서를 근거로 했는지 인용 가능.
4. 출처 SSE 페이로드에 `doc_id`, `filename` 포함.
5. 로그 라벨을 `[chat 3docs]` / `[chat all(N)]` 형태로 구분.

### 3.2 프론트엔드 — 범위 popover

`frontend/components/ChatPanel.tsx` 의 헤더에 “범위 · N개 문서” 드롭다운 버튼을 추가했다. 클릭하면 다음 popover 가 펼쳐진다.

- ready 상태 문서 목록을 체크박스로 표시 (파일명 + 카테고리 뱃지)
- 모두 선택 / 해제 단축 버튼
- “선택을 비우면 전체 문서를 대상으로 질의합니다” 안내
- 외부 클릭 / Escape 시 자동 닫힘 (`aria-haspopup="dialog"`, `aria-expanded` 부착)

상태 텍스트(`준비됨 · 선택된 3개 문서 기반`), 인풋 placeholder, 빈 상태 안내문이 모두 `0 / 1 / N` 케이스로 분기되어 사용자가 현재 범위를 즉시 인지할 수 있다.

### 3.3 UX 동선

| 상황 | 동작 |
|---|---|
| 좌측 리스트에서 문서 클릭 | 그 문서를 단일 선택으로 reset (의도 명확) + 대화 초기화 |
| 헤더 popover 에서 추가 체크 | 다중 선택으로 확장 |
| 모두 해제 | 자동으로 전체 모드 |
| 답변의 “출처” 카드 | 파일명 prefix 표시 — 어느 문서에서 인용했는지 시각적 확인 |

### 3.4 효과

- 6주차 과제 1.c 의 “전체 문서 대상 채팅” 요구를 충족하는 동시에, 사용자가 의도한 부분집합에 한정해 질의할 수도 있는 더 일반적인 형태가 됨.
- 단일 문서 모드는 정확히 “1개 선택” 케이스로 자연스럽게 흡수되어 기존 사용자 흐름을 깨지 않음.
- `top_k` 가 동일하면 다중 문서 모드에서 검색 노이즈가 늘 수 있어, 카테고리 필터(5주차 후속 과제) 와의 결합이 다음 과제로 자연스럽게 연결됨.

## 4. 백엔드 성능 개선

### 4.1 PDF 페이지 병렬 OCR 처리 (적용 완료)

#### 변경 전

`ocr.py`의 `_extract_pdf()`에서 PDF의 각 페이지를 순차적으로 OCR 처리. 페이지당 약 5~10초 소요되므로 10페이지 PDF 기준 **50~100초**가 걸렸다. 각 페이지의 OCR은 독립 작업이므로 순차 처리할 이유가 없다.

#### 변경 후

`asyncio.gather()`와 `asyncio.Semaphore`를 활용한 병렬 처리.

```python
sem = asyncio.Semaphore(MAX_CONCURRENT_PAGES)  # 기본 4

async def _ocr_page(idx, path):
    async with sem:
        text = await asyncio.to_thread(self._predict_image, path)
        return f"## Page {idx}\n\n{text}"

tasks = [_ocr_page(i, p) for i, p in enumerate(img_paths, start=1)]
page_texts = await asyncio.gather(*tasks)
```

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 10페이지 PDF 처리 시간 | ~50~100초 | ~15~30초 |
| 동시 처리 수 | 1 | 최대 4 (Semaphore) |
| GPU 메모리 부담 | 낮음 | 제한적 (조절 가능) |

`MAX_CONCURRENT_PAGES` 값은 GPU 여유에 따라 조정 가능하다.

### 4.2 청크 결과 캐싱 (적용 완료)

#### 변경 전

`GET /documents/{doc_id}/chunks` 호출 시마다 디스크에서 OCR 텍스트를 읽고 매번 청킹을 재수행.

#### 변경 후

업로드 처리 시점에서 청크 결과를 `.chunks.json` 파일로 저장하고, 이후 조회 시에는 캐시를 읽는다.

1. `_process_document()`: 청킹 완료 후 `{doc_id}.chunks.json`으로 저장
2. `GET /chunks`: 캐시 파일이 있으면 바로 반환, 없으면 재청킹 후 캐시 생성 (기존 문서 호환)
3. `DELETE /documents/{doc_id}`: 원본, .txt, .chunks.json 모두 정리

문서 내용이 변경되지 않으므로 캐시 무효화 이슈가 없다.

### 4.3 성능 개선 요약

| 항목 | 상태 | 영향도 |
|---|---|---|
| PDF 병렬 OCR | 적용 완료 | 멀티페이지 PDF 처리 속도 ~3배 향상 |
| 청크 결과 캐싱 | 적용 완료 | `/chunks` 응답 시간 대폭 감소 |
