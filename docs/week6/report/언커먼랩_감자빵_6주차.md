## 1. 버그 수정 — LLM 분류 필드 누락 대응

### 1.1 발생 현상

`GET /documents` 호출 시 500 에러가 발생했다.

```
pydantic_core._pydantic_core.ValidationError: 1 validation error for DocumentInfo
classification.reason
  Field required [type=missing, ...]
```

### 1.2 원인 분석

`Classification` 스키마(`backend/app/schemas.py`)는 `reason: str`을 필수로 요구하지만, qwen3:4b 모델이 분류 JSON을 생성할 때 `reason` 키를 누락하는 경우가 발생했다.

- **qwen3:4b 모델 한계**: 소형 모델은 JSON 스키마 키를 간헐적으로 누락. thinking 단계에서 "분류 완료"로 판단하면 부가 필드를 최종 출력에서 빼먹는 경향이 있음.
- **프롬프트 모호**: `CLASSIFY_PROMPT`에 `reason` 필드가 "필수"라는 명시가 없었음.
- `OllamaService.classify()`는 JSON 파싱 성공 시 dict를 그대로 반환 → 누락된 채 `_docs.json`에 저장됨 → 이후 `DocumentInfo(**d)` 변환 시 검증 실패.

### 1.3 적용한 수정

`backend/app/schemas.py` — `reason`을 `Optional`로 완화.

```python
class Classification(BaseModel):
    category: str
    category_code: str
    confidence: float
    reason: Optional[str] = ""   # ← 기존: reason: str
```

기존 저장된 문서도 정상 동작하며, 향후 LLM이 동일 패턴으로 누락해도 500이 아닌 정상 응답이 가능하다.

### 1.4 식별된 유사 위험 (후속 작업 권장)

| 위험도 | 항목 | 내용 | 권장 수정 |
|---|---|---|---|
| High | `filename=None` 저장 가능 | `UploadFile.filename`이 `Optional[str]` → `None`이면 list 호출 시 500 | `filename=file.filename or "unnamed"` |
| High | `classify()` 다른 키 누락 가능 | `category`/`category_code`/`confidence`도 빠질 수 있음 | `.get()`으로 모든 필수 키에 기본값 보강 |
| Mid | `vector_store.py` KeyError | `r.payload["text"]`가 `[]` 접근 → 외부 손상 시 500 | `.get("text", "")`로 통일 |
| Mid | `_docs.json` 스키마 진화에 취약 | 새 필드 추가 시 기존 문서 전부 깨짐 | 새 필드는 `Optional`로 시작 또는 검증 실패 문서 스킵 |

> 자세한 분석은 `docs/week6/bug-audit-2026-05-16.md` 참조.

## 2. 백엔드 성능 개선

### 2.1 PDF 페이지 병렬 OCR 처리 (적용 완료)

#### 변경 전

`ocr.py`의 `_extract_pdf()`에서 PDF의 각 페이지를 순차적으로 OCR 처리. 페이지당 약 5~10초 소요되므로 10페이지 PDF 기준 **50~100초**가 걸렸다. 각 페이지의 OCR은 독립 작업이므로 순차 처리할 이유가 없다.

#### 변경 후

`asyncio.gather()`와 `asyncio.Semaphore`를 활용한 병렬 처리.

```python
sem = asyncio.Semaphore(MAX_CONCURRENT_PAGES)  # 기본 4

async def _ocr_page(idx, path):
    async with sem:
        text = await asyncio.to_thread(self._predict_image, path)
        return f"## Page {idx}\n\n{text}"

tasks = [_ocr_page(i, p) for i, p in enumerate(img_paths, start=1)]
page_texts = await asyncio.gather(*tasks)
```

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 10페이지 PDF 처리 시간 | ~50~100초 | ~15~30초 |
| 동시 처리 수 | 1 | 최대 4 (Semaphore) |
| GPU 메모리 부담 | 낮음 | 제한적 (조절 가능) |

`MAX_CONCURRENT_PAGES` 값은 GPU 여유에 따라 조정 가능하다.

### 2.2 청크 결과 캐싱 (적용 완료)

#### 변경 전

`GET /documents/{doc_id}/chunks` 호출 시마다 디스크에서 OCR 텍스트를 읽고 매번 청킹을 재수행.

#### 변경 후

업로드 처리 시점에서 청크 결과를 `.chunks.json` 파일로 저장하고, 이후 조회 시에는 캐시를 읽는다.

1. `_process_document()`: 청킹 완료 후 `{doc_id}.chunks.json`으로 저장
2. `GET /chunks`: 캐시 파일이 있으면 바로 반환, 없으면 재청킹 후 캐시 생성 (기존 문서 호환)
3. `DELETE /documents/{doc_id}`: 원본, .txt, .chunks.json 모두 정리

문서 내용이 변경되지 않으므로 캐시 무효화 이슈가 없다.

### 2.3 성능 개선 요약

| 항목 | 상태 | 영향도 |
|---|---|---|
| PDF 병렬 OCR | 적용 완료 | 멀티페이지 PDF 처리 속도 ~3배 향상 |
| 청크 결과 캐싱 | 적용 완료 | `/chunks` 응답 시간 대폭 감소 |
