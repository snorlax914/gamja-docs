## **2. 회의 내용**

---

### **1. MVP 구현 현황 데모**

- 시연 가능한 수준까지 구현 완료
- 구현된 핵심 파이프라인
    - 회원가입 기능은 없음 (불필요)
    - 문서 업로드 → OCR → 청킹 → 임베딩
    - 질의는 RAG 기반 답변
- 데이터 저장 방식
    - 별도 DB 미사용 → 채팅 내용은 보존되지 않음
    - 문서는 서버에 저장
    - 문서 목록 / 데이터 불러오기도 서버 저장 기준
- 화면에서 확인 가능한 항목
    - 하단에서 Markdown OCR 결과 확인 가능
    - 진행 상황을 인디케이터로 확인 가능
- 외부에서 로그 확인은 불가능

- **처리 속도(현재)**
    - OCR: 페이지당 약 2초
    - 카테고리 분류: 약 40초
    - 청킹 / 임베딩: 빠름
    - 채팅 응답: 약 20~30초 (하드웨어 한계)
    - 모델 웜업 시 빠르게 종료되나, 초기 업로드 시 모델 로딩 시간 소요

- 논의: DB를 도입할 이유가 없었는지?
    - 현재 단계에서는 불필요한 측면 + 컴퓨터 메모리 문제

- 이슈: 채팅이 전체 문서를 대상으로 가져오도록 되어 있음 / 특정 페이지에서 서버 에러 발생 → 추후 별도 확인 예정

### **2. 아키텍처 구성**

- OCR: PaddleOCR, vLLM 으로 서빙
- LLM: Ollama + Qwen3:4B (하드웨어 제한으로 4B 사용)
- Vector DB: Qdrant 사용, 대시보드 제공으로 확인 가능하도록 준비 예정 (현재 포트 미개방)
- Embedding: BGE-M3, CPU로 구동되나 속도는 양호
- Backend: (FastAPI 등) API 서버 — 아직 API 수가 많지는 않음
- Frontend: Next.js, 단일 페이지 구성
    - 컴포넌트 분리: 업로드 영역 / 문서 영역 / 파이프라인 / 채팅 영역 분리

- 요청: 채팅 API가 실제 동작하는 모습을 확인하고 싶음

### **3. RAG Chunking 신규 구조 (과제 결과)**

- 청킹 관련 신규 구조 설계 완료
- 개선 내용
    - 표가 잘리는 부분을 Markdown으로 분할하여 LLM 인식률 개선
    - 홈페이지에서 청크 단위 확인 가능 (Markdown으로 분할된 상태)
    - 표도 가능한 한 하나의 청크에 들어가도록 개선
- 확인 사항
    - 청크가 잘린 것은 태그 문제가 아니라 500자(청크 사이즈) 한계 때문
    - 작은 표는 한 청크에 모두 포함됨
- HTML 태그 → Markdown 변환 도입 검토
    - OCR 모델 부분에서 별도 옵션 확인 필요
    - 옵션이 없으면 HTML → Markdown 변환 방식으로 신규 설계 진행

### **4. 헤더 주입(Header Injection) 전략 논의**

- 청크 앞에 의도적으로 헤더를 주입함
    - 이유: 청크만 떼어 보면 어떤 문서인지 알기 어렵기 때문
- 각 청크에 "헤더 + 페이지(2, 3 등)"를 넣은 이유 논의
    - OCR 결과가 그렇게 출력된 것으로 추정
    - 페이지 정보는 굳이 넣을 필요는 없으나, 특정 페이지를 지목해 질문하는 경우가 있어 필요
    - Vector DB 스키마 설계 시 메타데이터에 페이지를 넣는 등 다른 방식도 가능
- **결론: 헤더 일관성 부족 문제 → 모든 청크에 일관되게 넣거나, 다른 일관된 기준을 정해서 적용할 것**
- 추가 논의(형준 선배 정리 요청)
    - HTML을 Markdown으로 바꾸고 LangChain Splitter가 잘 동작하도록 하는 방안 확인
    - 인공적으로 헤더를 주입하는 방식의 의미가 큰지 / HTML 태그 표 청킹 시 헤더 추가가 유효한지 검토
    - 위아래로 긴 표는 잘리면 안 되므로 헤더 주입이 유효한 경우가 있을 것

### **5. 후속 과제 및 일정**

- 후속 과제: 하드웨어 한계로 기능 추가는 어려움 → 안정화 + UI 개선 중심
- 레포 정리: Docker로 들어가도록 폴더 구성하여 GitHub에 정리, 새 레포로 구성하면 확인 예정
- **전체 일정**
    - 6주차, 7주차 진행 후 다음 주 과제가 마지막
    - 8주차: 완성된 서비스 테스트 + 발표
    - 다음 주도 멘토가 시연할 수 있는 수준으로 다듬는 과제 부여
- DB 재검토 의견
    - DB를 넣어도 좋지 않을지 (Markdown OCR 데이터를 텍스트로 저장/불러오기가 좋지 않은 측면)
    - 성능이 받쳐주면 경량 DB(PostgreSQL 등) 도입 검토
